Distributed Memory System

How Product, Backend, Frontend, and QA teams share context and collaborate seamlessly using AI-SDLC's distributed semantic memory

Two Types of Memory

Understanding the difference between decision memory and code knowledge is key to using the system effectively.

🧠 Semantic Memory (Decisions)

Stores Decisions, rationales, QA notes, long-lived facts
Location .sdlc/memory/ — SQLite (local) + JSONL (team)
Good for "What did we decide about OAuth timeouts?"
Sync Git hooks export/import JSONL on commit/pull
Commands sdlc memory semantic-query, semantic-upsert

📦 Module KB (Code Facts)

Stores APIs, contracts, dependencies, layout from code
Location .sdlc/module/ — YAML/JSON (committed)
Good for "What endpoints does this service expose?"
Sync Hooks run sdlc module update on commit
Commands sdlc module init, module load, module validate

Key Benefits of Distributed Memory

No Status Meetings

"What did Backend decide about the API?" — Just query the shared memory. Cross-team log shows all decisions with ADO references.

New Engineers Productive Day 1

`sdlc module show` gives compact overview. `sdlc memory semantic-query` answers "why did we do it this way?" instantly.

Knowledge Survives Turnover

Decisions live in git, not employees' heads. When someone leaves, their context remains in version-controlled memory files.

Contextual AI Assistance

AI agents load relevant team memory automatically. "Implement the profile edit" — AI knows the API contract, design tokens, and known issues.

Audit-Ready Traceability

Every decision linked to ADO work items. Cross-team log shows who decided what, when, and why — compliance ready.

Efficient Token Usage

Load only what you need: `sdlc module load api` (2K tokens) vs full codebase (12K tokens). Budget enforced per role.

Summary

  • Team Memory: Each team (Product, Backend, Frontend, QA) maintains their own memory/team/ directory
  • Shared Memory: Cross-team decisions live in memory/shared/ with full ADO traceability
  • Git-Powered Sync: No separate infrastructure — memory flows through git hooks (export on commit, import on pull)
  • Two Storage Types: SQLite (local, gitignored) + JSONL (team bus, committed) for semantic memory; YAML/JSON files for module KB
  • Ask-First Access: AI queries memory before acting — "What do we know about this?" — ensuring informed decisions
  • Lifecycle Governance: Entries have states (active, superseded, archived) with retention policies and audit trails
